# UX Spec for Dev (RU)

## Цель
Дать разработчикам однозначную, технически пригодную спецификацию по интерфейсу: какие экраны есть, какие состояния, какие правила валидации и поведения.

## Когда использовать
- Уже есть карта сайта, прототипы или описанные пользовательские сценарии.
- Нужно подготовить единый документ для фронтенда/бэкенда.
- Нужно снять двусмысленности («а что если…?»).

## Структура спецификации

Для каждого экрана/страницы всегда описывай:

1. **Идентификатор и назначение**
   - ID страницы (например, `auth_login`, `campaign_dashboard`).
   - Краткое назначение: «страница авторизации», «дашборд кампании».

2. **Структура и блоки**
   - Перечень блоков сверху вниз.
   - Для каждого блока: название, краткое описание, обязательность.

3. **Поля и состояния**
   - Таблица: название поля, тип, формат, маска, обязательность, подсказка, текст ошибки.
   - Описать все состояния: нормально, фокус, ошибка, disabled, loading.

4. **Поведение и логика**
   - Что происходит при нажатии на кнопки.
   - Какие запросы выполняются (если известно).
   - Переходы между экранами.
   - Пограничные кейсы: пустые списки, длинные тексты, медленный ответ сервера.

5. **Адаптивность**
   - Ключевые брейкпоинты (мобильный, планшет, десктоп).
   - Что прячется, что складывается в меню, какие элементы меняют порядок.

6. **Тексты и состояния ошибок**
   - Конкретные формулировки сообщений пользователю.
   - Уровень строгости (ошибка/предупреждение/инфо).

## Правила оформления

- Пиши на русском, но ID и имена сущностей можно давать латиницей.
- Используй таблицы Markdown там, где есть поля и атрибуты.
- Ясно помечай то, что **обязательно к реализации**, и то, что «желательно, но можно отложить».

## Примеры запросов
- «Сделай UX-спеку для разработчиков по этим прототипам и описанию сценариев».
- «Оформи в виде спецификации все экраны админки для сайта кампании».
