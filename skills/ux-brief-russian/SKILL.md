# UX-brief Russian

## Цель
Помогать структурировать бриф от клиента и оформлять единый документ «Понимание задачи» для веб-проекта в информационном стиле, понятном всей команде.

## Когда использовать
- Клиент описывает задачу для сайта, лендинга, портала.
- Нужно превратить разрозненные сообщения, голосовые, ТЗ в один понятный документ.
- Нужно подготовить основу для карты сайта, UX-концепции и дальнейшего проектирования.

## Пошаговая инструкция для Claude

1. **Сбор информации**
   - Выпиши, что уже известно: цели, аудитория, продукт, ограничения, сроки, метрики.
   - Зафиксируй пробелы в информации как вопросы к заказчику.

2. **Структура документа «Понимание задачи»**
   Всегда соблюдай структуру разделов в таком порядке:
   1. Контекст и бизнес-цели
   2. Целевая аудитория и сценарии
   3. Продукт и ключевые предложения ценности
   4. Ограничения и допущения (технологии, сроки, бюджет, политика)
   5. Функциональные блоки и разделы сайта (без дизайна, только смысл)
   6. Требования к контенту (типы, объём, источники)
   7. Метрики успеха и аналитика
   8. Риски и открытые вопросы

3. **Правила языка**
   - Пиши короткими предложениями.
   - Используй информационный стиль: без воды, без оценочных суждений.
   - Избегай бюрократизмов и маркетингового тумана.
   - Все термины поясняй простым языком, как для старшеклассника.

4. **Формат вывода**
   - Выводи в Markdown с заголовками `##` и списками.
   - Отдельный блок «Вопросы к заказчику» — в конце документа.
   - Если данных не хватает, честно помечай: `_Нужна уточняющая информация_`.

5. **Проверка**
   - В конце добавь мини-чек-лист:
     - [ ] Цели понятны и измеримы
     - [ ] Аудитории и сценарии описаны
     - [ ] Основные разделы сайта перечислены
     - [ ] Риски и вопросы зафиксированы

## Примеры запросов
- «Собери из этой переписки документ "Понимание задачи" для лендинга кампании».
- «Оформи UX-бриф на основе этого голосового расшифрованного текста».
